## 一、核心基石论文精读

### 1.1 TimeChat-Online: 80% Visual Tokens are Naturally Redundant in Streaming Videos (arXiv:2504.17343)

本文对最新论文 **TimeChat-Online: 80% Visual Tokens are Naturally Redundant in Streaming Videos** 进行了深度剖析。

#### 核心思路 (Idea/Methodology)

TimeChat-Online 提出了一种受人类视觉“变化盲视 (Change Blindness)”现象启发的 **微分 Token 丢弃 (Differential Token Drop, DTD)** 模块，旨在解决流式长视频理解中的高冗余挑战：

- **纯视觉驱动的冗余过滤**：不同于以往依赖文本指令的压缩方法，DTD 在进入 LLM 之前通过比较相邻帧的像素级和特征级相似度，主动过滤掉静态或重复的 Token。

- **位置感知压缩**：利用多模态旋转位置编码 (M-ROPE) 在压缩前记录 Token 的 3D（时间、高度、宽度）位置，确保即使在大量 Token 丢弃后，模型仍能维持对时空定位和动作顺序的敏感性。

- **主动响应机制 (Proactive Responding)**：通过监控 Token 丢弃率的动态曲线（波谷代表场景剧烈切换），自动触发模型在关键时间点生成响应，而不仅是响应用户提问。

#### 实验设置 (Experimental Setup)

- **基座模型**：基于 **Qwen2.5-VL-7B** 架构，支持高 FPS 的密集帧处理。

- **数据集**：构建了 **TimeChat-Online-139K** 指令微调数据集，其视频平均时长为 **11.1 分钟**（涵盖 5 分钟到数小时），包含回溯追踪、实时感知及前瞻响应任务，并包含 20K “不可回答”负样本以训练模型学会等待未来信息。

- **微调策略**：冻结视觉编码器，对 Projector 和 LLM 进行全参数微调。采样率保持在 1 FPS，最大上下文支持扩展至 1016 帧（压缩后约 6K Token）。

#### 实验结果 (Results & Performance)

- **极高的压缩效率**：在流式视频任务中，DTD 成功削减了 **82.8%** 的视频 Token，且在 StreamingBench 上维持了 98% 的性能，响应延迟缩减至 1.76x。

- **长视频推理表现**：在处理 **30-60 分钟** 的超长视频测试集（如 VideoMME Long Subset）时，模型减少了 **84.6%** 的 Token，且准确率反而提升了 **5.7%**。这证明了在长上下文中，减少冗余 Token 不仅是效率优化，更是对“视觉噪声”的有效过滤。

### 1.2 LLaVA-Mini: Efficient LMM with One Vision Token (arXiv:2501.03895)

LLaVA-Mini 突破了传统 MLLM 对大量 Vision Tokens 的依赖，实现了“1图1Token”的极端压缩。

- **核心思路 (Idea/Methodology)**：
  - **模态预融合 (Modality Pre-fusion)**：研究发现 Vision Tokens 仅在 LLM 的前几层起关键作用（用于将视觉信息融合进 Text Tokens）。LLaVA-Mini 在进入 LLM 之前引入轻量级的 Transformer 块进行预融合，从而允许将输入 LLM 的视觉 Token 压缩至 **1个**。
  - **查询驱动压缩 (Query-based Compression)**：利用可学习查询（Learnable Queries）和交叉注意力机制，在预融合后对视觉特征进行池化与压缩。

- **实验设置 (Experimental Setup)**：
  - **基座模型**：Vicuna-7B / LLaMA-3.1-8B。
  - **数据量**：558K 预训练数据 + 665K 指令微调数据。
  - **应用场景**：支持高分辨率图像（672px）和超长视频处理。

- **实验结果 (Results & Performance)**：
  - **极端效率**：仅用 1 个 Token 即可在 11 个图像和 7 个视频 Benchmark 上达到甚至超过 LLaVA-v1.5（576 Token）的性能。
  - **资源节省**：FLOPs 降低 **77%**，推理速度提升 **2.92 倍**，单图显存占用从 360MB 降至 **0.6MB**，支持在 24GB 显存上处理 **10,000+ 帧**（3 小时+）视频。

## 二、 视频 MLLM Token Reduction 相关研究现状

目前的研究路径可分为 **Training-free (训练无关)** 与 **Fine-tuning (微调引入)** 两大类，其优劣势对比如下：

### 2.1 Training-free 路径（强行丢弃/池化）

- **代表工作**：AdaptToken (基于信息熵选择)、VisionZip (基于特征空间压缩)。

- **优势**：即插即用，无需昂贵的微调成本。

- **痛点**：
  - **缺乏语义适应性**：简单的均匀采样或特征聚类容易丢失细粒度、非显著但关键的动作 Token（如快速运动物体）。
  - **时空信息受损**：强行池化（Pooling）或合并（Merging）往往会模糊原始的空间坐标，导致模型无法处理“左侧物体是什么”或“动作发生了几次”等定位与计数任务。

### 2.2 Fine-tuning 路径（引入微调）

- **代表工作**：VideoLLM-MoD (深度混合模型)、Resampler-based 架构。

- **现状**：目前微调方法多集中在 Projector 层的特征对齐，或者引入固定的 Resampler 将每帧压缩为固定数量。

- **痛点**：
  - **静态压缩比瓶颈**：固定 Token 输出无法适应视频内容动态变化的本质（静态场景需极低 Token，复杂动作需高采样）。
  - **训练成本高昂**：在长视频场景下进行端到端微调的数据量和算力需求极高。
  - **缺乏指令交互性**：现有的压缩逻辑往往与用户的特定 Query 脱节，无法做到“按需压缩”。

## 三、领域前沿工作全景梳理 (Literature Review 2025-2026)

长视频多模态大模型 (Video MLLM) 在理解、推理和生成任务中展现了卓越的能力，但其巨大的计算与内存开销成为了实际应用，尤其是实时流式场景下的核心瓶颈。问题的根源在于视频数据内在的高度冗余性：一个长视频由数千甚至数万帧构成，其中包含大量的时空重复信息（如静态背景、缓慢变化的动作）。这些帧被视觉编码器（如 ViT）处理后，会产生海量的视觉 Token 序列。当这些序列被送入大语言模型 (LLM) 时，LLM 中自注意力机制的二次方复杂度 ($O(N^2)$) 会导致推理延迟急剧增加，同时，生成的键值对缓存 (KV Cache) 也会迅速膨胀，耗尽内存资源。

为应对这一挑战，学术界在 2025-2026 年间集中探索了多种 Token Reduction 技术，旨在不显著牺牲模型性能的前提下，大幅削减输入到 LLM 的视觉 Token 数量或压缩其中间表示。这些前沿工作大致可归纳为四大技术范式：**即插即用/免训练加速 (Plug-and-Play / Training-free)**、**层次化/金字塔压缩 (Hierarchical Compression)**、**流式/在线压缩 (Streaming/Online Compression)**，以及多种方法共有的 **空间与时间联合路由 (Spatio-Temporal Routing)** 策略。本章将对这些领域的代表性工作进行系统性梳理与评析。

### 3.1 即插即用/免训练加速 (Plug-and-Play / Training-free)

免训练（Training-free）或即插即用（Plug-and-Play）的方法是当前研究的主流，其核心优势在于无需对庞大的 Video MLLM 进行成本高昂的重新训练或微调，即可作为独立模块嵌入现有模型以实现推理加速。这类方法通常基于启发式规则或轻量级计算来评估和筛选 Token，具有极高的灵活性和通用性。

**代表性工作：**

- **Video Compression Commander (VidCom2, EMNLP 2025)**
  - **核心思想**：该工作认为，以往方法统一压缩所有帧的方式忽视了不同帧在视频中的重要性差异。关键信息可能集中在少数“独特”的帧中，而大量帧是冗余的。因此，`VidCom2` 提出了一种 **基于帧独特性（Frame Uniqueness）的自适应压缩策略**。
  - **压缩机制**：
    1. **帧压缩强度调节 (Frame Compression Adjustment)**：首先，通过计算每个帧的视觉 Token 与整个视频的全局表征之间的相似度，来量化该帧的“独特性分数”。独特性越高的帧被认为包含更多关键信息。随后，根据此分数动态地为每一帧分配合理的 Token 预算（即保留率），确保独特帧得到更精细的保留，而冗余帧则被更激进地压缩。
    2. **自适应 Token 压缩 (Adaptive Token Compression)**：在获得每帧的预算后，进一步评估每个 Token 的“综合独特性”，该独特性同时考虑了其在 **帧内 (intra-frame)** 的局部显著性和在 **视频间 (inter-video)** 的全局新颖性。最终，根据综合分数和预算，保留每帧中最重要的 Token。
  - **优缺点**：
    - **优点**：**高度自适应**，能够根据内容动态分配计算资源，在极高的压缩率下依然能有效保留关键视觉信息。作为即插即用模块，**兼容性强**，可与多种 Video MLLM 和高效的注意力算子（如 Flash Attention）结合。实验表明，在仅保留 25% 视觉 Token 的情况下，性能可达原始模型的 99.6%，并将 LLM 生成延迟降低了 70.8%。
    - **缺点**：该方法主要针对离线（offline）视频处理，对于需要严格因果约束的流式场景，其依赖全局信息的帧独特性评估机制需要调整。

- **KiToke (arXiv 2026)**
  - **核心思想**：`KiToke` 将 Token 压缩问题建模为 **全局多样性最大化 (Global Diversity Maximization)**。它跳出了局部或分段处理的框架，通过 **核密度估计 (Kernel Density Estimation, KDE)** 在整个视频的 Token 流上评估冗余，从而做出更全局最优的剪枝决策。
  - **压缩机制**：
    1. **全局冗余度量**：利用基于核函数的冗余度量，为所有 Token 计算一个平滑的、非参数化的密度。高密度区域意味着信息冗余，低密度区域则代表着独特的、需要保留的信息。这种方法能够同时抑制空间（帧内）和时间（帧间）的重复。
    2. **区间感知 Token 合并 (Interval-aware Token Merging)**：为了在压缩后依然维持视频的动态时序结构，`KiToke` 引入了一种轻量级的时间区间构建方法，将具有相似动态的连续 Token 分组，并在这些区间内进行合并，有效避免了因孤立地删除 Token 而破坏时间连贯性。
  - **优缺点**：
    - **优点**：**在极端压缩率下表现优异**，由于其全局视角，即使在保留率低至 1% 的情况下，依然能比依赖局部启发式的方法保留更多信息。免训练，且与具体模型和 Tokenizer 解耦，**通用性好**。
    - **缺点**：全局核密度估计的计算开销可能随视频长度增加而变大，在极长视频或低延迟流式场景下的效率有待进一步验证。

- **ForestPrune (arXiv 2026)**
  - **核心思想**：`ForestPrune` 创新性地使用 **时空森林建模 (Spatial-Temporal Forest Modeling)** 来进行高比例的视觉 Token 压缩。它将视频帧中的 Token 组织成树状结构，从而对视频内容形成整体性的理解，并基于此进行剪枝。
  - **压缩机制**：
    1. **构建 Token 森林**：根据 Token 间的语义、空间和时间约束，跨视频帧构建“Token 森林”。在森林中，语义相似且时空上邻近的 Token 被连接成树。
    2. **评估节点重要性**：基于树的深度和节点在树中的角色（如根节点、叶节点）来评估每个 Token 树和其中节点的重要性。通常，深度更深、作为关键连接点的节点被认为更重要。
    3. **全局剪枝决策**：综合所有 Token 的重要性评分，做出全局最优的剪枝决策，保留最重要的 Token。
  - **优缺点**：
    - **优点**：**实现了极高的压缩率**，在仅保留 10% Token 的情况下，性能仅下降 3.2%，优于许多同类方法。其建模方式能够很好地捕捉视频内容的结构和连续性。同样是免训练方法。
    - **缺点**：构建森林和评估重要性的过程涉及图算法，其计算复杂度可能较高，在推理时会引入一定的额外开销，需要权衡其与压缩带来的总收益。

- **VidKV (arXiv 2025)**
  - **核心思想**：此工作另辟蹊径，不直接减少 Token 数量，而是通过 **极低比特量化来压缩 LLM 的键值对缓存 (KV Cache)**，从而大幅降低推理过程中的内存占用。
  - **压缩机制**：
    1. **混合精度量化 (Mixed-Precision Quantization)**：`VidKV` 发现 VideoLLM 的 Key Cache 和 Value Cache 具有不同的分布特性。因此，它对 Key 采用 1.5-bit 量化（通过 FFT 变换稳定分布），对 Value 采用 1.58-bit 量化。
    2. **按通道量化 (Per-Channel Quantization)**：与传统 LLM 中按 Token 量化的方法不同，`VidKV` 指出 VideoLLM 的 Value Cache 在通道维度上变化更大，因此采用按通道量化能更好地保留性能。
    3. **语义 Token 过滤**：结合了 Token 筛选机制，对语义上更重要的视觉 Token 在量化时进行特殊保护，以实现精度和压缩率的更好平衡。
  - **优缺点**：
    - **优点**：**极大降低了内存占用**（可达 80%），对于处理长视频或大批量推理至关重要。作为即插即用模块，无需训练。
    - **缺点**：主要关注点是内存而非计算延迟。虽然减少了数据搬运量，但对 LLM Prefill 阶段的计算量削减有限。其性能高度依赖于对 VideoLLM 内部数值分布的精确假设。

> 这些即插即用的方法共同推动了 Video MLLM 的效率边界，它们的核心设计哲学在于 **通过更精细化的信息评估机制（如独特性、多样性、结构重要性）取代过去粗暴的均匀采样或简单合并，从而在大幅降低数据量的同时，最大程度地保留了对理解视频至关重要的视觉线索**。

### 3.2 层次化/金字塔压缩 (Hierarchical Compression)

层次化方法利用视频固有的“分段-帧-区域”或模型内部的“浅层-深层”结构，在不同粒度上逐步进行信息压缩与抽象。这种多尺度处理方式有助于在早期阶段剔除粗粒度的冗余，在后期阶段保留精细的关键细节。

**代表性工作：**

- **HieraVid (arXiv 2026)**
  - **核心思想**：`HieraVid` 明确提出一个三级层次化剪枝框架，该框架充分利用了视频的 **分段-帧 (segment-frame)** 结构和 LLM 内部 **单向多模态信息流** 的特点。
  - **压缩机制**：
    1. **段级别 (Segment-level)**：首先在时间上将视频划分为多个片段 (segment)，并在每个片段内进行初步的空间合并，减少初始 Token 数量。
    2. **帧级别 (Frame-level)**：在同一片段内，识别并联合剪枝内容相似的帧，以保留片段内的视觉多样性。
    3. **层级别 (Layer-level)**：在 LLM 的不同层之间进行渐进式剪枝。利用 LLM 浅层主要处理局部、低级特征，而深层处理更抽象、语义信息的特性，逐层增加剪枝力度，动态地减少冗余。
  - **优缺点**：
    - **优点**：**结构化地利用了视频和模型的双重层次**，使得剪枝决策更具信息效率。分层处理允许在不同阶段采用不同的策略，比单一的输入层剪枝更精细。实验表明，保留 30% 的 Token 即可达到 SOTA 性能，并保持了超过 98% 的原始模型表现。
    - **缺点**：三级剪枝的策略设计和超参数调整较为复杂。层级间的剪枝需要对 LLM 的内部结构进行一定的侵入式修改，虽然是免训练，但实现上比纯输入端处理的方法更具挑战。

- **Accelerating Streaming Video Large Language Models via Hierarchical Token Compression (STC, arXiv 2025)**
  - **核心思想**：`STC` 是一个专为 **流式视频** 设计的即插即用层次化压缩框架，它同时优化了 **ViT 编码** 和 **LLM Prefill** 两个阶段，以应对流式场景下的双重瓶颈。
  - **压缩机制**：
    1. **STC-Cacher (ViT 阶段)**：在 ViT 编码阶段，通过缓存和重用时间上相似帧的特征来减少冗余计算。它仅对包含新信息的“动态 Token”进行计算，而对“静态 Token”（如背景）则直接重用缓存的特征，且这一过程严格遵守因果性，不依赖未来帧。
    2. **STC-Pruner (LLM Prefill 阶段)**：在 Token 进入 LLM 之前，通过一个“双锚点”机制评估每个 Token 的新颖性。一个锚点是 **时间上下文锚点 (TCA)**，代表历史信息的积累；另一个是 **空间上下文锚点 (SCA)**，代表当前帧的全局信息。只有那些同时与历史和当前帧全局信息都“不相似”的 Token 才会被保留。
  - **优缺点**：
    - **优点**：**首个同时优化 ViT 和 LLM 两个主要瓶颈的流式压缩框架**。`STC-Cacher` 显著降低了视觉编码的延迟，而 `STC-Pruner` 有效缩短了 LLM 的处理序列。即插即用，且在流式基准测试中表现出色，在 ReKV 框架上减少了 24.5% 的 ViT 延迟和 45.3% 的 LLM Prefill 延迟，同时保持 99% 的准确率。
    - **缺点**：缓存机制 (`STC-Cacher`) 引入了额外的内存管理开销。双锚点剪枝 (`STC-Pruner`) 的性能依赖于历史信息 buffer 的大小和更新策略，这在极长视频中可能成为新的挑战。

- **Global Compression Commander (GlobalCom2, AAAI 2026)**
  - **核心思想**：该工作虽然最初是为高分辨率图像设计的，但其思想已扩展到视频领域。它利用了高分辨率图像/视频通常被处理为“一个低分辨率的全局缩略图 + 多个高分辨率的局部裁块”这一层次结构，提出 **用全局信息指导局部信息的压缩**。
  - **压缩机制**：
    1. **全局指挥官**：将低分辨率的全局缩略图视为“指挥官”，评估各个局部裁块的信息丰富度。
    2. **自适应压缩**：对于信息量更丰富的裁块，分配更多的 Token 预算；反之，对于冗余或与全局信息重叠的裁块，则进行更大力度的压缩。这种“全局到局部”的引导避免了对所有局部细节一视同仁的盲目剪枝。
  - **优缺点**：
    - **优点**：**有效解决了多视角/多分辨率输入的冗余问题**，符合人类视觉系统先整体后局部的感知模式。通过全局上下文指导，可以减少因过度压缩局部信息而产生的“幻觉”。
    - **缺点**：该方法更适用于将视频分解为全局和局部视图的模型架构，对于采用统一帧表示的模型，其适用性需要进一步改造。

> 层次化压缩方法的核心贡献在于 **将视频的内在结构或模型处理流程的阶段性特点融入到压缩策略中**。相比于“扁平”地处理所有 Token，这种方法通过多尺度、分阶段的剪枝，能够实现更高效、更智能的资源分配。

### 3.3 流式/在线压缩 (Streaming/Online Compression)

流式处理是长视频理解的关键应用场景，如实时监控分析、直播内容理解等。该场景对算法提出了两大严苛挑战：**因果性 (Causality)**，即处理当前帧时不能使用未来帧的信息；以及 **累积性 (Accumulation)**，即处理过程中信息不断累积，导致计算和内存开销随时间线性增长。流式压缩方法专为解决这些问题而设计。

**代表性工作：**

- **StreamingTOM: Streaming Token Compression for Efficient Video Understanding (arXiv 2025)**
  - **核心思想**：`StreamingTOM` 是一个严格遵守因果性约束的免训练、两阶段流式压缩框架，它同时解决了 LLM 处理前 (pre-LLM) 和处理后 (post-LLM) 的瓶颈问题。
  - **压缩机制**：
    1. **因果时间规约 (Causal Temporal Reduction, CTR)**：在 **pre-LLM 阶段**，该模块为每一帧设置一个固定的 Token 预算。它仅根据当前帧和紧邻的前一帧之间的变化以及 Token 本身的显著性来选择保留哪些 Token。这使得每帧的 Prefill 成本从与所有历史 Token 相关降低到仅与固定预算相关，实现了可预测的低延迟。
    2. **在线量化内存 (Online Quantized Memory, OQM)**：在 **post-LLM 阶段**，为了控制无限增长的 KV Cache，OQM 将历史 Token 以 4-bit 的低精度格式存储。在需要时，根据相关性按需检索和解量化，从而使活跃的 KV Cache 大小保持有界，与视频流长度无关。
  - **优缺点**：
    - **优点**：**首个在流式场景下实现严格因果的 pre-LLM Token 压缩** 的免训练方法，解决了以往方法只关注压缩 KV Cache 而忽略 Prefill 成本的痛点。两阶段设计全面优化了流式推理的效率，实现了高达 15.7 倍的 KV Cache 压缩和 2 倍的“首个 Token 生成时间” (TTFT) 加速，且内存峰值更低。
    - **缺点**：固定的每帧 Token 预算可能在面对视频内容复杂度剧烈变化时不够灵活。4-bit 量化虽能大幅节省内存，但仍可能在某些对精度极度敏感的任务上造成性能损失。

- **Accelerating Streaming Video Large Language Models via Hierarchical Token Compression (STC, arXiv 2025)**
  - **核心思想**：`STC`（已在上一节介绍）同样是为流式场景设计的领先工作。其 `STC-Cacher` 和 `STC-Pruner` 组件在设计上都严格遵守了“未来不可知”的原则。
  - **流式特性**：
    - `STC-Cacher` 通过与前一个 **参考帧** 比较来识别动态 Token，这是一个典型的因果操作。
    - `STC-Pruner` 依赖于历史信息构成的 **时间上下文锚点 (TCA)** 和当前帧的 **空间上下文锚点 (SCA)**，整个决策过程不依赖任何未来信息。
  - **在流式场景下的优势**：`STC` 提供了一个优雅的解决方案，在不破坏时间因果性的前提下，同时降低了上游 ViT 的计算冗余和下游 LLM 的序列长度。**其层次化的设计使得它在处理流式数据时，既能利用短期缓存减少重复计算，又能通过长期历史锚点把握内容演变，实现高效且稳健的在线压缩**。

> 流式压缩技术的核心在于 **设计出既能有效评估信息重要性，又严格遵守因果约束的在线算法**。`StreamingTOM` 和 `STC` 从不同角度给出了出色的答案：`StreamingTOM` 采用“固定预算 + 量化内存”的硬约束策略，保证了资源占用的可预测性；而 `STC` 则采用“缓存 + 双锚点”的动态评估策略，力求在效率和灵活性之间取得平衡。

### 3.4 空间与时间联合路由 (Spatio-Temporal Routing)

这一范式可以看作是前述多种方法在实现 Token 选择时的共性核心技术，即如何智能地在庞大的时空数据场中“路由”计算资源，将它们引导至信息最密集的区域。这些方法通过联合分析空间（帧内）和时间（帧间）的线索来决定哪些 Token 值得被保留和处理。

- **VidCom2 (EMNLP 2025)** 的路由策略体现在其 **双重独特性评估** 上。它不仅评估一个 Token 在当前帧内的独特性（空间维度），还评估其相对于整个视频的独特性（时间维度），实现了时空联合路由。

- **STC (arXiv 2025)** 的 `STC-Pruner` 组件通过 **时空双锚点** 机制实现路由。空间锚点捕捉当前帧的静态信息，时间锚点捕捉历史动态，一个 Token 只有在时空两个维度上都表现出新颖性时才被选中。

- **ForestPrune (arXiv 2026)** 则通过构建 **时空森林** 来进行路由。Token 间的连接关系同时包含了空间上的邻近性和时间上的连续性，使得重要性评估天然地融合了时空信息。

- **KiToke (arXiv 2026)** 使用 **全局核密度估计**，其度量空间是所有 Token 特征组成的联合分布。因此，其密度评估自然地反映了一个 Token 在整个时空数据场中的冗余度，实现了隐式的时空路由。

> 这些方法共同表明，**孤立地考虑空间或时间维度的信息都是不够的**。长视频 Token Reduction 的关键在于设计出能够 **高效建模时空联合分布、并从中识别出信息量最大子集的路由策略**。无论是基于独特性、新颖性还是全局多样性，其本质都是在寻找一种低成本的代理指标 (proxy) 来近似 Token 的真实重要性，从而指导计算资源的最佳分配。

### 3.5 长视频场景的特殊挑战与文献洞察

上述前沿工作在提升 Video MLLM 推理效率方面取得了显著进展，但针对 **长视频（例如数十分钟甚至数小时）** 的特殊挑战，仍存在一些开放性问题：

1. **跨段叙事一致性与长期依赖 (Cross-segment Narrative Coherence & Long-term Dependencies)**：长视频通常包含复杂的故事情节和跨越很长时间的因果链。现有的压缩方法，特别是那些基于局部窗口或固定大小历史缓冲区的（如 `STC`、`StreamingTOM`），可能会在信息流被截断或过度压缩时丢失关键的早期线索，导致模型无法理解长期依赖关系。例如，一个在视频开头出现但在后续长时间未再出现的物体，其信息可能会被逐渐“遗忘”。`ForestPrune` 和 `KiToke` 的全局建模思想为此提供了一个有潜力的方向，但其在超长序列上的可扩展性仍需验证。

2. **上下文维护与缓存策略 (Context Maintenance and Caching Strategy)**：在流式处理中，如何高效地管理和更新不断增长的上下文至关重要。`StreamingTOM` 的在线量化内存 (OQM) 是一种有效的解决方案，但其固定的内存大小可能在信息密度极高的视频片段中成为瓶颈。`STC` 的缓存机制 (`STC-Cacher`) 主要针对 ViT 编码阶段的短期冗余，而 LLM 层面 KV Cache 的有效管理仍是一个难题。`VidKV` 提出的极低比特量化是一种纯内存优化的方案，但它没有解决序列本身过长导致的计算问题。未来的研究需要将 Token 剪枝与更智能的 KV Cache 替换/压缩策略（如基于注意力分数的缓存驱逐）更紧密地结合起来。

3. **时空分辨率与信息密度的权衡 (Trade-off between Spatio-Temporal Resolution and Information Density)**：长视频不仅时间跨度大，其空间和时间的事件密度也可能极不均衡。例如，一段长时间的静态风景后可能突然出现一个短暂但关键的动作。`VidCom2` 基于帧独特性的自适应预算分配机制部分解决了这个问题，但它主要在帧级别进行调整。如何在更细粒度的 Token 级别或更粗粒度的场景级别动态调整时空采样率和压缩比，以匹配视频内容的“信息速率”，是一个更具挑战性的问题。

4. **视频-文本跨模态同步成本 (Cost of Video-Text Cross-modal Synchronization)**：在 LLM 内部，视觉 Token 和文本 Token 之间的交叉注意力是理解的关键，也是计算的重点。Token Reduction 方法虽然减少了视觉 Token 的数量，但可能会破坏原始的视频-文本对齐关系。例如，`HieraVid` 提出的层级剪枝虽然高效，但需要精心设计以确保在剪枝过程中多模态信息流不被错误地中断。如何设计一种压缩算法，使其既能减少 Token 数量，又能显式地保留或重构对跨模态理解至关重要的时空结构，是未来值得探索的方向。

## 四、 核心学术动机与破局点：从“静态极致压缩”到“时空双重弹性感知”

### 4.1 现有路线的困境：时空信息的不可兼得与“空间坍塌”

当前长视频 Token Reduction 技术在提升长上下文处理效率上取得了显著进展，但面临着一个根本的学术悖论：**如何以极低的算力成本，保留回答细粒度 Query 所需的空间细节？**

1. **不可逆的时序丢弃 (Streaming Temporal Drop)**：如 TimeChat 采用的微分丢弃策略，通过硬截断移除冗余帧提升效率。但多轮交互中，早期帧的丢弃会导致模型遭遇“视觉失忆”，无法回溯。

2. **极致的空间压缩 (Extreme Spatial Compression)**：部分前沿工作尝试将每帧特征极度压缩（如压缩至 1-Token）。虽然保留了完整的时间轴，但引发了严重的 **空间坍塌 (Spatial Collapse)** 问题。1-Token 的强制池化/均值化抹平了画面的高维空间细节，导致模型可以理解宏观剧情，却对“左下角的微小物体”、“屏幕上的文字（OCR）”等细粒度问题彻底失效。

### 4.2 本文的学术主张：基于 Query 驱动的弹性时空上下文

为了打破上述僵局，本文提出一种全新的正交联合压缩范式——**时空双重弹性压缩 (Spatiotemporal Elastic Compression)**。真正的长视频理解应当是“按需索取”的。我们在时间轴上进行静态去冗余，在空间轴上则摒弃“一刀切”的固定压缩比，设计了一种 **动态级联呼吸式架构 (Cascaded Breathing Architecture)**：系统将高分辨率特征（如 16-Token）作为底层 Buffer 驻留显存，而向 LLM 浅层输入极简的区域锚点（如 4-Token 以保留基础象限空间信息）。在 LLM 前向传播的中间层，文本的真实意图 (Query) 被用来作为路由信号，精确唤醒并按需扩容（4-Token -> 16-Token）关键帧的高清细节。这既避开了全高清输入的 OOM（显存溢出）灾难，又完美保留了长视频中的微小局部空间信息。

## 五、 算法创新：级联特征金字塔与“意图驱动”的无缝扩容

为了实现上述构想，同时避免动态增删 Token 带来的 RoPE 位置编码混淆和序列长度突变等工程灾难，我们的算法包含以下四个优雅的执行阶段：

### 5.1 阶段一：构建轻量级视觉特征金字塔 (Cascaded Visual Pyramid)

原始视频通过 ViT 提取每帧的高清特征（如 576 Token）后，立即通过一个轻量级的级联压缩器（例如基于 Token Merging 或简单 MLP 的池化网络），生成多粒度特征金字塔：

- **Level-0 (Discard)**：576 Token，提取完成后即时丢弃，不进入后续网络。

- **Level-1 (Detail Buffer)**：**16 Token**，作为保留细粒度局部空间信息的高清 Buffer 驻留于 GPU 显存。对于 1 小时视频（约 3600 帧），16 Token 仅占用不到 6 万 Token 的显存量，工程上极为轻量且安全。

- **Level-2 (Base Anchor)**：**4 Token**，作为基础视觉锚点。4 个 Token 天然对应画面的左上、左下、右上、右下四个象限，能在极低维度下保留最基本的空间方位与显著物体坐标。

### 5.2 阶段二：时间轴静态过滤 (Temporal Pruning)

在进入 LLM 之前，利用连续帧之间 4-Token 级别的相似度或运动显著性（Motion Saliency），执行高效的时间帧筛选。将原始的 $T$ 帧筛选为最具信息量的 $K$ 帧，从时间维度剥离静态冗余背景（如冗长的静态会议画面）。

### 5.3 阶段三：浅层极简感知与意图酝酿 (Coarse Prefilling)

在 LLM 的前 $M$ 层（如前 1/3 的层数），网络仅接受选出的 $K$ 帧 * 4-Token 作为视觉输入，与文本 Prompt 一起进行高效计算。在这一阶段，模型以极低的算力成本快速摸清视频的“剧情大纲”，并在文本 Token 的隐状态（Hidden States）中酝酿出用户的高阶语义意图。

### 5.4 阶段四：中层弹性扩容与精准路由 (Elastic Spatial Routing)

在 LLM 的中间层引入 **Routing Gate（路由门控）** 与 Cross-Attention 机制，实现按需的高清特征唤醒：

1. **意图匹配**：当前文本 Token 的 Hidden States 作为 Query，与 $K$ 个 4-Token 锚点进行匹配打分。

2. **弹性扩容 (4 -> 16)**：当判定某几帧包含回答用户问题所需的核心细节时，触发局部扩容操作。

3. **隔空取物**：为了保持主干序列长度不变，文本 Token 直接通过 Cross-Attention 从 GPU 显存的 Level-1 Buffer 中读取命中帧的 **16-Token 高清特征**，精准吸收细粒度信息并融入后续的深层推理中。

## 六、 实验设计与验证规划 (Experimental Setup)

为了严格验证“弹性时空压缩”在保留细粒度空间细节及处理长上下文时的优越性，我们将跳出传统偏向宏观理解的粗粒度榜单，重点在以下高度依赖细粒度空间解析和长程检索的 Benchmark 上进行评估：

### 6.1 验证防“空间坍塌”能力 (Anti-Spatial Collapse)

- **评估数据集**：**EgoSchema** (真实第一视角高难度 QA) 与 **Video-MME (OCR & Visual Detail 子集)**。

- **验证逻辑**：由于第一视角视频中的细微物体交互以及屏幕文本极易在极度压缩中丢失，我们将证明本文的 4 -> 16 Token 弹性扩容机制，能够在总体算力持平的情况下，大幅提升文本可读性与局部物体识别准确率。

### 6.2 验证超长上下文的细粒度检索 (Long-Context Fine-grained Retrieval)

- **评估数据集**：**MLVU (Needle QA 子任务)** 与 **LongVideoBench**。

- **验证逻辑**：通过绘制“上下文长度 vs. 检索准确率”以及“性能 vs. 显存消耗”的热力图（Pareto Frontier），证明在面对长达一小时的视频时，本方法依托极低的 Baseline Token（4-Token）和轻量的显存 Buffer（16-Token），能精准保住“大海中的针”，且彻底杜绝 OOM 风险。
