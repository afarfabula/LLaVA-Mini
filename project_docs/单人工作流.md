# 单人工作流

## 1. 目标

本文档定义本项目在 solo 模式下的工作方式。

这里的 solo 模式不是“一个人随便做实验”，而是：

- 训练主线由单一负责人推进
- 重要决策必须能从文件中追溯
- 一段时间后回来仍然能恢复上下文

## 2. 适用范围

当前范围：

- 这台机器只负责训练和训练侧验证
- 评测暂时不在这台机器上做
- 当前优先目标是让训练修改具备可复现性

## 3. 工作规则

### 3.1 一个主入口

- 训练必须只有一个主启动入口
- 临时调试命令可以存在，但不能替代主入口
- 如果主入口变化，必须同步更新 `任务计划.md` 和 `训练运行手册.md`

### 3.2 不允许静默改动

- 影响训练逻辑、模型结构、数据格式、显存行为的改动，必须记录到 `实验进展.md`
- 没写入文档的关键决策，不算完成

### 3.3 将学术改动和工程改动分开

- 学术改动：压缩策略、路由逻辑、特征金字塔、时序剪枝
- 工程改动：路径、脚本、环境、命名、日志、checkpoint 管理
- 日志中不要把两者混写而不加区分

### 3.4 小步验证

- 在第一次 GPU 跑通前，不要同时堆太多未验证改动
- 先验证 import、启动、前反向、张量形状和显存
- 之后再讨论性能提升或下降

### 3.5 仓库文档是事实来源

- 聊天记录不是事实来源
- 临时记忆不是事实来源
- 仓库中的文档和脚本才是事实来源

## 4. 完成定义

一个任务只有同时满足以下条件才算完成：

- 代码或文档已经落到仓库中
- 启动路径仍然清晰可读
- 结果已经写入 `实验进展.md`
- 下一步动作对未来的自己是明确的

## 5. 文档职责

- `原始研究动机.md`：保留研究问题的原始展开
- `研究摘要.md`：保留稳定的研究主线
- `任务计划.md`：维护当前阶段与下一步
- `训练运行手册.md`：维护训练机上的执行方式
- `实验进展.md`：维护时间线上的真实进展
