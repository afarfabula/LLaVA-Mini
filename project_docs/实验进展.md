### 进展 011

- 标签：`单卡修复`
- 目标：优先跑通单卡训练链路，并进一步解释 DDP 失败原因
- 改动：
  - 修正 `llavamini/train/train.py` 中数据集异常重试的随机索引上界，避免重试时再次越界
  - 修正 `llavamini/train/llava_trainer.py`，单进程场景下直接使用原始 DataLoader，绕开 `accelerate.prepare(dataloader)` 返回空 batch 的问题
  - 修正 `scripts/train_solo_sft.sh`，支持 `SINGLE_PROCESS=1` 的单进程单卡启动方式
  - 在 `GPUS=1` 且未显式指定 `CUDA_VISIBLE_DEVICES` 时，默认固定到 `CUDA_VISIBLE_DEVICES=0`
  - 新增 `scripts/inspect_smoke_batch.py`、`scripts/run_singlecard_smoke.sh`、`scripts/run_singlecard_smoke_to_file.sh` 以便定位和复现单卡 smoke
- 结论：
  - 之前的 `NoneType` 报错并不是模型前向本身坏掉，而是小规模 smoke 数据在当前 worker 上进入 Trainer 后出现了“batch 被吃空”的问题
  - 关键原因之一是单进程运行时看到 2 张可见 GPU，Trainer 将 train batch size 视作 `per_device_train_batch_size * n_gpu`，配合 `drop_last=True` 导致 1 条 smoke 数据被直接丢空
  - 固定 `CUDA_VISIBLE_DEVICES=0` 后，单卡路径已经可以再次进入 `Loading checkpoint shards` 和训练 step
  - DDP 路径的核心阻塞仍是 `NCCL` 初始化时报 `CUDA driver version is insufficient for CUDA runtime version`
- 当前状态：
  - 单卡路径已明显比之前更深，已越过参数解析、数据集初始化与空 batch 问题
  - 但本轮尚未在输出目录中看到 `checkpoint-1`，仍需继续确认 worker 会话中断后训练进程是否真正完成保存
